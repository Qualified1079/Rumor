# Rumor — Claude Code context

## Architecture at a glance

Three Gradle modules with strict layering:

```
:core       — pure JVM/Kotlin. No Android imports. All protocol, crypto, scheduling logic.
:app        — Android. Depends on :core. Room DB, ViewModels, Compose UI, transport.
:simulator  — JVM CLI. Depends on :core. Ktor dashboard, in-memory repos, sim transport.
```

Key invariants:
- `GossipEngine.relay()` never touches blocklist state — only `emitToInbox()` does.
- `TrustLevel.BRIDGED` messages are never re-relayed. Bridge traffic cannot forge VERIFIED trust.
- `BRIDGE_UNSIGNED` sentinel is honored **only** for `MessageSource.LOCAL_BRIDGE`; peer transport always verifies Ed25519.
- `trafficClass` is derived, never on the wire — a sender cannot claim INFRASTRUCTURE class for a bulk payload.
- Size ceilings enforced in `trafficClass`: INFRA/REALTIME=16 KB, TRANSFER_SETUP=256 KB. Oversized → BULK.
- **MAC addresses are never identity.** Android randomises MACs per-connection and they are trivially spoofable. The only identity is the userId proven via HELLO Ed25519 challenge-response. `WifiP2pDevice.deviceAddress` is only used as an opaque connection target for the OS — never as a cache key, cooldown key, or trust anchor.

## Coding conventions

- No CDN or external assets. All UI is offline-first (no WebView phoning home).
- F-Droid compatible: no proprietary SDKs, no Play Services.
- Minimal comments — only write WHY, not WHAT. No docstrings unless a contract is genuinely non-obvious.
- No premature abstractions. Three similar lines beats a helper. No feature flags.
- Room schema changes: bump `RumorDatabase.version`. Uses `fallbackToDestructiveMigration()` in dev. Current version: **4**.

## DI (Koin) wiring

Single source of truth: `app/src/main/java/com/rumor/mesh/di/AppModule.kt`.
Simulator uses constructor injection with in-memory repos (`simulator/…/data/InMemoryRepos.kt`).

When adding a new core class that needs to reach the app:
1. Add `single { … }` to AppModule.
2. Add an in-memory stub to `InMemoryRepos.kt` if the simulator needs it.
3. If it's a new `ContactRepository`/`RouteRepository` method, update all four impls:
   - `ContactRepositoryAdapter` / `RouteRepositoryAdapter` (Room)
   - `InMemoryContactRepository` / `InMemoryRouteRepository` (simulator)

| O12 | **`deterministic-replay` residual non-determinism** | `SimTransport.exchange()` was fixed to (a) drop real `delay()` calls and (b) use the tick's seeded RNG. Two variance sources remain: (1) `GossipEngine` runs internal handlers on `Dispatchers.Default`; coroutine scheduling order isn't guaranteed (affects `totalMessages` by ~3–5%). (2) `ConcurrentHashMap` iteration order in `knownMessages()` determines which RNG values the conditioner consumes; drop counts vary by <0.1% across runs. Both are tolerated by ±5% relative bounds on both messages and drops in the assertion. If variance grows, the escalation path is a single-threaded dispatcher for `SimNode.scope` (to fix handler order) and sorting `knownMessages()` by ID (to fix conditioner RNG consumption order). | O11 | **`no-bridged-rerelay` assertion may need escalation** | The scenario runner's `NoBridgedRerelay` check is a post-hoc snapshot scan: at the end of a scenario it walks every node's stored messages and flags any BRIDGED-trust message whose `recipientId` doesn't match that node's `userId`. It's reliable in the simulator because the in-memory `MessageRepository` never evicts during a run, but it would miss a violation where the engine relayed a bridged message, then the message expired or got evicted before the scan. If the in-sim message store ever starts trimming during a scenario — or if this assertion graduates to production telemetry — escalate to a live observer: add a `relayObserver: ((RumorMessage) -> Unit)?` hook to `GossipEngine.relay()` and have the runner subscribe per-node. | O13 | **BloomFilterData OOM on adversarial expectedItems** | A peer could send a bloom filter claiming `expectedItems = Int.MAX_VALUE`, causing the BloomFilterData constructor to attempt a multi-GB allocation and crash the process. Hard caps are the wrong fix — in 50 years a large bloom may be completely legitimate. Right fix: graceful degradation. Wrap `BloomFilterData` construction in a try/catch for `OutOfMemoryError` and `IllegalArgumentException`; on failure, fall back to sending up to `MAX_OFFER_PER_EXCHANGE` messages without bloom filtering (same as the no-prior-knowledge path). Receiver keeps running, just slightly over-offers this exchange. |
| O14 | **Compressed-archive extraction UI** | There is no compression at the protocol layer — `Chunker` splits raw bytes, no zip/gz. So a zip bomb at protocol level is not possible; the hash is of the full uncompressed data and is covered by Ed25519, so a sender cannot lie about it without breaking their signature. The risk is purely application-layer: a sender transmits a 1MB `.zip` payload which the receiver extracts to 100GB. Fix: before any extraction of a compressed MIME type, show a non-dismissable dialog: "Compressed archive received — Download: XMB / Estimated extracted: YGB." Do not extract until user confirms. This is a UI feature, no protocol changes needed. |
| O15 | **Runtime plugin loading (DEX loader)** | Current plugins (Meshtastic, MeshCore) are compiled into the APK — not loadable at runtime. For a true mod scene, plugins need to be droppable files. Two mechanisms that compose: (1) **DEX loader**: a `.rumor` bundle (zip of `.dex` + a manifest) loaded via `DexClassLoader` at runtime — the mod-scene primary path, distributable via flash drive or chunked transfer over the mesh. (2) **Bound-service APK**: a separately installed APK exposes a bound service; Rumor binds to it by intent — good for plugins that need their own Android permissions (camera, contacts, location) since those are scoped to the plugin APK, not Rumor's process. Both paths feed into the same `PluginContext` interface. The DEX path runs in Rumor's process with Rumor's permissions; the bound-service path runs isolated. No central repository; the mesh itself is the distribution channel. **Pre-ship requirement**: a static manifest declaration in the plugin bundle listing which `PluginContext` APIs it calls, shown to the user as a disclosure popup before enable — not a security control (a malicious plugin can lie), but informed consent. |
| O16 | **Relay-path spam: per-sender rate limiting** | A malicious plugin can call `injectFromPlugin` or `composeDirect` at arbitrary rate. Round-robin in the Scheduler ensures one spammer cannot monopolize bandwidth — their messages compete fairly. Blocking is a display filter only (relay path never sees blocklist — intentional, see design decisions). The residual risk is CPU and `MessageStore.ingest` pressure from high-rate senders even for blocked users, since ingest (sig verify + dedup + write) runs before the blocklist check. Mitigation: add a per-`senderId` token bucket in `MessageStore.ingest` — if a sender exceeds N ingests/sec, skip ingest (not just inbox) for that sender for the remainder of the window. This does NOT affect relay to others (relayed copies bypass the ingest gate); it only limits how much CPU/storage one sender can consume on this specific node. Threshold should be generous (e.g. 100/sec) to avoid false positives on high-traffic legitimate nodes. |
| O17 | **Extend bridged content beyond direct peers (`BRIDGE_VOUCHED` trust level)** | Today BRIDGED traffic is dropped on relay (see design decisions), so a bridge only reaches its direct Rumor peers — Charlie three hops away in the Rumor mesh never sees Bob's Meshtastic message even though Alice's bridge has it. Proposed mechanism: introduce a third trust level between VERIFIED and BRIDGED, called `BRIDGE_VOUCHED`. Bridge wraps the foreign message in a Rumor-signed outer envelope that explicitly carries `{originNetwork, originSenderId, originSignatureIfAny, payload}`. The outer envelope is signed by the bridge's Rumor key — but the wire format and trust level make it unambiguous that the bridge is vouching for *delivery* only, NOT for content authenticity. Other Rumor peers can relay it because the outer envelope verifies. Display layer renders as "via Alice from meshtastic:Bob" with a distinct visual treatment from native Rumor messages. Recipients decide per-bridge whether to trust that bridge's vouching (UI: enable/disable bridge-vouched relay per known bridge). Replaces the current absolute "never relay BRIDGED" with a more nuanced "never relay BRIDGED without an authenticated outer envelope from a bridge node we know about." Scope: new TrustLevel enum value, new wire envelope format, signature verification of outer envelope at every receiver, UI for per-bridge trust toggles, three-state visual rendering. Real design work; not started. |
| O18 | **Transfer-control UI (pause / resume / cancel)** | `TransferAssembler` and `TransferSender` have no user-facing controls. Once a large transfer starts, the user can't cancel it without quitting the app, and it can't be paused if the device is going to be off-network temporarily. Add: cancel button (drops the transfer record, cancels watchdog, sends a CANCEL message so sender stops resending chunks), pause button (suspends watchdog and incoming chunk processing without dropping state), resume (re-arms watchdog from current state). UI should show current progress (`receivedIndices.size / totalChunks`), throughput, and estimated time remaining. No protocol changes for cancel/pause if implemented as receiver-local; sender-side cancel needs a new `TRANSFER_CANCEL` message type so the sender stops queuing chunks. |
| O19 | **Pre-send file manipulation (compression, format conversion)** | Pure app/UI layer, no protocol changes. Before chunking and sending a file, offer the user: image compression (JPEG quality slider, downscale to N pixels), video transcode (bitrate selector via Android's `MediaCodec`), archive options for multi-file selections (zip / tar.gz). Show "Original: XMB → Compressed: YMB" before confirm. Reduces bandwidth and storage cost across the mesh significantly for image-heavy traffic, which is the dominant case in practice. Independent of O14 (extraction-warning popup) but shares the same UI surface. |
| O20 | **Private key in Android Keystore (TEE-backed)** | Currently the Ed25519 private key lives in app storage — readable by any code running in Rumor's process, including DEX plugins. Migrate to Android Keystore so the key never leaves the Trusted Execution Environment. `PluginContext` exposes `signWithLocalKey(bytes): ByteArray` only — no raw key access. Pre-ship requirement before any plugin loader ships. Note: Keystore-backed keys are device-bound and non-exportable; backup/restore of identity requires a separate identity-backup protocol (seed phrase or similar). |
| O21 | **TOFU key pinning and identity hardening** | `userId` is already `SHA-256(publicKey).toHex()` (see `CryptoManager.publicKeyToUserId`), so the userId↔pubkey binding is cryptographic — a sender cannot claim a userId that doesn't hash from their pubkey, and every receiver can re-verify cheaply. The real identity-hardening gap is **display**: a 64-char hex userId is unreadable, and human-set display names can be freely spoofed (two people can both call themselves "Bob"). Solution: derive a short visual fingerprint from the pubkey — take 8 bytes of `SHA-256(publicKey)`, map each byte to an emoji from a curated set of ~256 visually distinct emoji (animals, food, objects, vehicles; no faces, no flags, no near-duplicates, all cross-platform-stable). Display as `Bob [🐅🍧🚲🌵]`. Four emoji = 32 bits of fingerprint, collision-resistant up to ~65k contacts, easy to compare at a glance. Full hex userId revealed only on tap-through. **Pin display-name → userId binding** in `ContactRepository`: if a previously-seen userId arrives with a different display name, surface "Bob changed their display name to Robert" rather than silently updating. Pinning is for display-name churn, not key churn (latter is impossible by construction). Display names should be entirely optional — users who want maximum clarity can show only the emoji fingerprint. |
| O22 | **Native delayed / recurring message scheduling** | Reduces pressure on plugins needing outbound-message capability. UI: compose a message, tap a clock icon, set send time or recurrence (hourly / daily / on next peer contact). Stored locally; `MeshService` fires them on schedule. Covers the primary legitimate use case (status bots, scheduled announcements) without requiring a plugin that declares outbound-message capability. |
| O23 | **User-defined storage quota and plugin storage scopes** | Before first message, setup flow asks user to allocate a storage budget (e.g. "Reserve 2GB for Rumor"). All repos (messages, chunks, transfers, plugin data) share this budget. Plugins receive a scoped storage allocation within the global budget; if a plugin needs more than its default slice it requests an increase via a system prompt the user approves — same pattern as Android runtime permissions. File access for plugins follows the Storage Access Framework picker model (same as Signal): plugin declares which file types it needs, user grants access to specific files or folders, not blanket storage. |
| O24 | **Plugin security tiers: DEX (read-only) vs APK (dangerous)** | DEX `.rumor` bundles run in Rumor's process and are limited to read-only / display-layer `PluginContext` APIs (observe incoming, register display filters, open settings panel). Any plugin declaring dangerous capabilities (compose outbound messages, mutate contacts, register DM envelopes, open network connections) must ship as a bound-service APK running in its own isolated Android process with its own Android permission grants. Loader enforces this: DEX plugins requesting dangerous APIs are rejected at load time. User sees "this plugin requires installing a companion app" for APK-tier, which is natural friction proportional to actual risk. Manifest capability list re-disclosed on plugin update if capabilities changed. |
| O25 | **Plugin crash isolation: auto-disable on unhandled exception** | All calls into plugin code (every `PluginRegistry.dispatch*` and every `PluginContext` callback) are wrapped in try/catch. On unhandled exception: log the stack trace, disable the plugin for the session, surface a notification "Plugin X crashed and was disabled." Do not propagate the exception to the host app. Persistent crash (disabled across restarts) if it crashes on init. Plugin author can subscribe to crash reports via a declared crash-report endpoint in the manifest. |
| O26 | **Community attestation and peer plugin sharing** | No central plugin repository. Plugin publishers broadcast a signed metadata record on the mesh (`pluginId`, `version`, name, description, capability list, `SHA-256(bundle)`, size). Reviews are **separate** signed messages anchored to `(pluginId, bundleHash)` — they propagate as ordinary mesh content, not chunked with the bundle, so the bundle itself stays small. A "plugin directory" view aggregates: shows plugins seen on the mesh, queries reviews on demand. Acquiring a plugin from a direct peer: chunked transfer of the bundle bytes, addressed as DM to the requester. Acquiring a plugin you've heard about but no direct peer has: broadcast `PLUGIN_REQUEST(pluginId, version)`; anyone holding it replies with either the bundle (chunked DM) or a metadata response "I have v5 and v7, not v6 — want those instead?" Requester chooses. Versioning: every bundle carries `(pluginId, version)` in its manifest; old versions remain valid (no forced upgrades). Authors push new versions by broadcasting a new metadata record; users may subscribe to "notify me when new versions of plugin X appear." Reviews are pseudonymous (userId-signed) so a reviewer's track record across plugins is visible — a reviewer with consistently useful reviews becomes a trustable voice without any central authority. |
| O27 | **Deanonymization limits: honest documentation + future mitigation options** | Relay batcher provides timing-correlation resistance at the protocol level. The ephemeral nature of the mesh (no persistent central server logging correlations) improves on centralized systems. But in a fixed physical post-disaster network where the same devices are near the same nodes daily, graph analysis of relay patterns still leaks social graphs even without content. A local malicious plugin observes everything including timing, peer identities, and message rates. **Ship-blocker:** honest documentation — Rumor provides better privacy than centralized apps but weaker anonymity than Tor; do not overclaim in user-facing copy. **Future mitigation options to keep on the table, none planned today:** (1) **Cover traffic** — constant-rate dummy messages so an observer can't distinguish active from idle nodes; expensive on battery and bandwidth. (2) **Onion routing layered on the mesh** — messages wrapped in N layers of encryption, each hop peels one; resistant to traffic analysis, adds latency. (3) **Random delay-padding at relay** — stronger relay batcher with variable delays that actively decorrelate, not just batch. (4) **k-anonymity batching** — hold messages until N from different senders can be emitted together, mix-style. All four are real engineering work and have battery/bandwidth/latency costs; keep the door open but require demonstrated need before implementation. |
| O28 | **Wire-parser fuzzing pass** | Before public release, run automated fuzzers against every parser that touches untrusted bytes: `Json.decodeFromString` for every wire format, `fromBase64`, `BloomFilterData.deserialize`, `Chunker.reassemble`, every protobuf path in the Meshtastic/MeshCore bridges. Tool: Jazzer (JVM fuzzer compatible with libFuzzer). Each parser gets a fuzz harness; CI runs them for N minutes per PR. Crashes are bugs, not "interesting findings." Most parsers already use the `runCatching { ... }.getOrNull() ?: return` pattern which catches well-formed exceptions, but fuzzing catches the cases that pattern misses: OOM on adversarial sizes (related to O13), stack overflow on deeply nested JSON, integer overflow on length fields. Plugin-scene threat model makes this urgent — a malicious plugin can craft any payload and broadcast it, looking for crashable peers. Plugin lowers the barrier from "build custom Rumor binary" to "drop a 50-line plugin." |
| O29 | **DM breadcrumb routing** | DMs are currently flooded with bounded TTL=15 — `recipientId` rides as plaintext on every hop and bandwidth scales linearly with mesh size per DM. Replace with per-hop next-peer scoring against a weighted blend of signals: (1) **reverse path from recent inbound from X** — strongest because it's proof of reachability, not guess; (2) **DM-ACK return path** — confirmed working route; (3) **outbound delivery success** — handed to P, no flood-fallback observed in window; (4) **`TopologyTracker.overlapFor(X)`** — already collected for P2 batch-shaping; (5) **per-peer "recent senders" bloom** exchanged in handshake — if P's bloom contains X, P recently held X's messages; (6) **inbound hop-distance distribution** — average effective hops to X per arrival-peer; lowest wins; (7) **priority-peer declarations** — already local; (8) **route advertisements in handshake** — see O31; (9) **opt-in contact-graph hints**; (10) **co-presence history** — peers commonly in same physical area as X at similar times, pure local inference; (11) **online-presence beacons** — O30. All signals decay by recency. Per-hop decision: weighted score per candidate peer, hand to top-K (K=2–3) for robustness, not top-1. Cold-recipient fallback: if no peer scores above confidence threshold, immediately flood — don't iteratively waste hops. Loop safety: hard hop ceiling (~30) regardless of routed/flooded split. |
| O30 | **Online-presence beacons** | Short-TTL INFRASTRUCTURE messages "I saw X within last N minutes" gossiped at low frequency. Lets peers far from X learn coarse reachability without polling. Feeds O29's signal 11. Aggregated with recency decay. Privacy trade-off: leaks "X was recently online and visible to someone in this mesh segment" — same leak class as DM `recipientId` today. Should be capped (peer only beacons for contacts it has marked, not for every userId ever seen) and rate-limited so a malicious plugin can't flood. |
| O31 | **Route advertisements in HELLO** | Each peer includes a small `recentlyExchangedWith: List<UserId>` (top-N, e.g. 20) in HELLO. Cheap (~640 bytes), lets neighbours immediately know if a path through this peer is plausible. Goes inside the signed HELLO transcript so it can't be spoofed by MITM. Recipient uses for O29 scoring. Default-on for users explicitly marked as priority-peer (they've already opted into being routing nodes); default-off for everyone else. Per-contact opt-out ("don't list me in your route advertisements") for users who don't want this leak. |
| O32 | **TTL split: `routedHops` vs `floodedHops`** | DMs travelling along a confirmed O29 breadcrumb shouldn't burn TTL — only flood-mode hops should. Replace single `hopsToLive` with two counters: `floodedHops` decrements only when the relay falls back to flood mode (no peer scored above threshold); `routedHops` tracks total path length for diagnostics. Hard ceiling on `routedHops + floodedHops` (~30) caps loops regardless of mode. Backwards-compat: a v0.1 client sees only `hopsToLive` and decrements every hop — safe degradation, just less efficient. The two-counter field lives in `_ext` (per report 10 wire-format invariants) until a protocolVersion bump. |
| O33 | **Battery profile and "protest-day" budget** | Measure continuous BLE + Wi-Fi Direct scan + relay power draw on representative devices (Pixel-class, mid-range Samsung, 4-year-old budget). Target envelope per research: 8 hours active use on ≤50% battery; 12–24 hours unattended on a modern phone with conservative duty cycling. Tune `SCAN_MODE_LOW_POWER` defaults, motion-aware throttle (AirGuard pattern), hard cap on reconnect attempts to vanished peers (Meshtastic #4444 / #2659 failure mode), exponential discovery backoff when no peers seen (G2 generalized). Declare `connectedDevice` foreground-service type (Android 14+) with `FOREGROUND_SERVICE_CONNECTED_DEVICE` permission. First-run flow requests `REQUEST_IGNORE_BATTERY_OPTIMIZATIONS` — without it Samsung/Xiaomi/Huawei kill the service regardless of FGS status. Without measured numbers, every claim about field viability is unsupported. |
| O34 | **Honest threat-model page (Briar-style)** | Single user-facing page modelled on briarproject.org/manual/. State explicitly what Rumor protects against: passive eavesdropping at protocol layer (X25519+AES-GCM), MITM (Ed25519 sigs), server compromise (none exists). State explicitly what Rumor does NOT protect against: traffic analysis by a co-located observer with multiple radios (sybil-equipped adversary at ~$1000), malicious plugins running on the user's device, compromised endpoints, BLE/Wi-Fi adjacency leak of userId, a state actor with deep-packet capability targeting an individual. Bridgefy's reputational collapse came specifically from marketing privacy properties it did not deliver — the lesson is to undersell, never oversell. Pre-ship blocker. Pairs with O27 and replaces vague "honest documentation" language with a concrete deliverable. |
| O35 | **Pre-crisis contact onboarding flow** | The single most consistent failure mode across every studied deployment (Bridgefy HK 2019, India CAA 2019, BLM 2020, FireChat Iraq 2014) is "installed during the crisis, has no contacts, abandoned within hours." Crisis-install never works. Design low-friction onboarding *before* crisis: (a) address-book scan with explicit per-contact consent to share a Rumor invite; (b) QR-code exchange that works in-person without internet (already partly designed via O21 fingerprints); (c) invite-link over existing channels (Signal, SMS, email) that bootstraps a Rumor contact without revealing identity to the carrier — link encodes pubkey + display name + optional one-time secret for SAS-style verification on first connect; (d) "import contacts from Signal" companion flow if technically feasible. Without a pre-existing contact graph, every other Rumor feature serves an empty inbox when it matters most. |
| O36 | **No global broadcast room — design decision recorded** | Rumor has no global "public chat" / "town square" room and will never add one. FireChat's defining feature became its defining failure: public rooms at protest density get poisoned by state-injected or merely incoherent content within hours, signal/noise collapses, the app becomes unusable exactly when needed. Rumor's broadcast traffic class is for *messages from contacts you've chosen to follow*, not for an open firehose. The temptation to add a "discovery" or "town square" surface will recur in feature requests; the answer is in the deployment record. Group messaging (future, per discussion above) is *closed-membership* groups only, never open. A **local-only display filter** — "show me broadcasts from non-contacts in my immediate area" — is a UI feature and is fine to ship; it does not change protocol behaviour, the network carries broadcast traffic the same way regardless. What is forbidden is a network-level "open room" feature where the protocol distinguishes "public" traffic from contact-graph traffic and relays the former with different semantics. |
| O38 | **Receiver-side forward secrecy via rotating recipient prekeys** | Current DM crypto (ephemeral sender X25519 + recipient static X25519 + AES-GCM + Ed25519 outer sig) provides **sender-side FS only**. A relay node holding stored ciphertext can decrypt every past DM the moment the recipient's static key leaks — equivalent to the original Briar-BHP gap (CVE-2023-33982), and Briar got away with it only because Tor sat underneath. We have no Tor layer. Fix: recipient periodically generates short-lived X25519 prekeys `R_t` signed by their long-term Ed25519 key, gossips them as small INFRASTRUCTURE messages; senders DH against the freshest `R_t` they hold rather than the long-term `R`; recipient deletes `R_t_priv` after expiry. Stored ciphertext encrypted to an expired `R_t` is unrecoverable even with the long-term key. Document offline-for-weeks fallback honestly: a recipient who's been offline longer than any cached prekey is reachable only via the long-term key with documented FS regression for that case. Privacy by structure, not by code-comment — the private key actually does not exist anymore once deleted, which is what makes the property real against an adversary with full bit-awareness. |
| O39 | **Per-message key lifecycle audit** | Verify ephemeral X25519 private, HKDF intermediate, and AEAD key are destroyed immediately after use on both sender and recipient. If `composeDirect` retains the ephemeral or the derived AEAD key in any cache, or if DM decrypt regenerates the AEAD key on demand from `(senderEphemeralPub, recipientStaticPriv)` rather than decrypting once and storing plaintext (encrypted at-rest under a separately-rotated storage key), even sender-side FS is lost. Pre-ship audit: trace the per-message key path through `composeDirect`, `MessageStore.ingest`, and the DM decrypt helper; confirm zeroize-on-exit; add a test that the ephemeral private is unrecoverable after compose. Sig-scope check: Ed25519 signature must cover ciphertext + AD, never plaintext (Bridgefy got this wrong — surrogate-signature / UKS attacks). |
| O40 | **Relay-deletion-on-ACK** | Even with O38/O39 the FS-exposure window for stored ciphertext is "until the message expires off all relays." Tighten: on recipient ACK, sender (or recipient) gossips a signed `DELETE(messageId, recipientUserId)`; relays purge from their MessageRepository on receipt. Doesn't recover FS against an attacker who already exfiltrated the ciphertext, but bounds the storage window for *future* compromise — a relay seized 30 days later only holds messages no recipient has ACK'd yet. Compatible with existing dedup machinery (the deleted messageId stays in the seen-set to prevent re-ingestion). Cost: one extra INFRASTRUCTURE message per ACK'd DM; affects BROADCAST messages too if we extend the pattern. |
| O41 | **Identity rotation message + UX** | Because `userId = SHA-256(pubkey)`, rotating an X25519/Ed25519 keypair today produces a new userId — every contact has to re-add you, contact-graph continuity is lost. This makes the only PCS mechanism (manual rotation after compromise) so painful nobody will use it. Fix: signed `IDENTITY_ROTATION(oldUserId, newUserId, newPublicKey, signature_by_old_key)` message broadcast on the mesh; recipients holding `oldUserId` as a contact verify the signature (proves continuity of control), update the contact record to `newUserId`, surface a notification "Bob rotated their key — same person, new identity hash" so display-name pinning (O21) doesn't fire a false alarm. Pairs with O38: routine prekey rotation is automatic; long-term key rotation is rare but must be possible without nuking the contact graph. Coercion case: the old key signing a rotation message is the proof — an attacker who has only stolen ciphertext but not the old key cannot forge this. |
| O42 | **Replace bloom-filter offer/want with Range-Based Set Reconciliation (Negentropy)** | Current gossip uses a bloom filter sized for expected message-set size; on long partition heal the actual symmetric difference can blow past the parameters, causing false-positive collapse (receiver thinks peer has things it doesn't, peer holds back, messages don't flow). Existing graceful-degradation fallback (O13) handles this safely but inefficiently. Port Negentropy (Meijers 2023, arXiv 2212.13567; reference impl github.com/hoytech/negentropy, MIT licensed, ~500-line Kotlin port): O(d log N) bandwidth, deterministic, no parameter tuning, no decode-fail mode, deployed in Nostr's strfry at tens-of-millions-of-elements scale via NIP-77. Operates on `(timestamp, id)` tuples — matches existing message model. Composes with peer-aware batch shaping (P2) — the overlap heuristic stays; only the wire format changes. Solves O13's BloomFilter OOM problem as a side effect. Don't roll your own; don't pick CPISync (cubic compute), classic IBLT (parameter tuning fragility), or rateless IBLT (too new, no battle-tested impl). |
| O43 | **Per-link heal-storm token bucket with traffic-class priority** | Round-robin scheduling guarantees fairness across senders but does not bound total throughput on partition heal. When two large clusters merge, every node tries to flush its delta to every newly-reachable peer; aggregate is still N×peers messages. Add per-outbound-link token bucket sized by traffic class: INFRASTRUCTURE / REALTIME pre-empt the bucket, BULK and reconciliation backlog share what's left. Combined with O42's lazy-pull-by-default (small fingerprint exchange before pushing bodies), this addresses both bandwidth spikes and the DTN-documented "storage exhaustion during long partition" failure mode. |
| O37 | **Wire-format invariants — pre-ship lock-in** | Once Rumor ships to phones that may never update (rural users, disaster-cached APKs), the wire format is law. Six invariants must land before v0.1: (1) **Single `WireJson` instance** in `:core` with `ignoreUnknownKeys = true`, `encodeDefaults = true`, `explicitNulls = false`; all wire I/O routes through it; grep-fail any other `Json {}` in CI. Today `GossipSession.kt:63` has the only correctly-configured instance — needs to be promoted to `:core/WireJson.kt` and reused everywhere. (2) **Reserve `_ext: Map<String, JsonElement>? = null`** on every top-level wire object (`GossipPacket`, `RumorMessage`, `Hello`, `Offer`, `BloomFilterData`, `TransferMetadata`, `BlocklistPublish`). v0.1 ignores it; v0.2+ carries additive fields here without bumping `protocolVersion`. Costs ~2 bytes when absent, buys infinite forward-compat headroom. (3) **Relay forwards original received bytes, never re-encoded JSON.** JSON loses unknown fields on re-encode, which silently strips v0.2 fields from v0.1 relays and breaks both forward compat and signature verification. Audit `GossipEngine.relay()` to confirm. (4) **Domain-separated signatures** — every Ed25519 sig is computed over `"rumor-<msgtype>-v<n>:" || canonical_bytes`. `BlocklistPublish` already does this (`blocklistSignableBytes` with `rumor-blocklist-v1:`); extend the pattern to every signed message, HELLO transcript, introduction, presence beacon. Prevents cross-protocol replay when v2 arrives. (5) **Version negotiation lives inside the signed HELLO transcript.** `protocolVersion`, `maxProtocolVersion`, `supportedFeatures: List<String>` go into the HELLO challenge-response signed bytes — TLS-1.3 downgrade lesson: an unauthenticated version field is MITM-tamperable. Session uses `min(mine, theirs)`. (6) **Field names are append-only**; reused names = silent data corruption between versions. Keep a `RENAMED_FIELDS_NEVER_REUSE.md` listing every deprecated name forever. Deprecation policy: ≥2 years of parse-and-relay support for any retired version (rural and disaster-cache users are the design target — they're exactly the population that won't update). Plan for any v0.1 client to still be on the mesh in 2050 (IPv6 lesson). Activation rule: never gate features on peer-quorum signalling (BIP9 lesson — adversarial peers will refuse to signal). Frame size cap stays at 256 KB regardless of version. |

### Design decisions recorded

- **Latency NOT used for routing.** On BLE/Wi-Fi Direct, measured latency is mostly discovery timing. Ranking is by `bytesRelayed DESC, sessionCount DESC`. `latencyMs` stored for diagnostics only.
- **Relay path never sees blocklist.** Blocking is a display filter only — nodes relay everything regardless of blocklist state. This is intentional: if nodes stopped relaying blocked senders, a coordinated block could erase a person from the entire mesh, turning individual preferences into network-level censorship. Charlie needs Bob's emergency broadcast even if Alice blocked Bob. Your device's resources serve the mesh, not your social preferences. The residual CPU/storage cost of ingesting high-rate blocked senders is addressed separately by per-sender rate limiting (O16).
- **`BLOCKLIST_PUBLISH` is TRANSFER_SETUP, not INFRASTRUCTURE.** Full snapshots can exceed 16 KB; only incremental `BLOCKLIST_DIFF` stays at INFRASTRUCTURE tier.
- **Bridge traffic never re-relayed (today).** Re-signing a bridged message with the bridge's Rumor key would make every foreign-network message look like the bridge node originated it, and would let the bridge launder content it never received. Current invariant: BRIDGED → no relay, full stop. **Cost:** the bridge only reaches its direct Rumor peers; Charlie three hops away in the Rumor mesh never sees Bob's Meshtastic message. This is more restrictive than the design needs to be — see O17 (BRIDGE_VOUCHED trust level) for the path to extending bridged-content reach without re-introducing the laundering risk.
- **Bridges today are broadcast-only.** Both current bridge implementations (Meshtastic, MeshCore) bridge BROADCAST traffic only; DM bridging is unimplemented (O5). The DmEnvelope framework (G5/O5a) supports encrypted DM bridging via Architecture B (envelope passthrough) — the framework is in place; the per-bridge envelope code is what's missing. For BROADCASTs, Rumor hands the radio plaintext and the radio applies channel encryption with its own PSK; inbound, the radio gives us plaintext via `MeshPacket.decoded` (Meshtastic) or after firmware decryption (MeshCore). We never touch channel keys for broadcasts.
- **Encrypted bridging — two architectures, pick one before designing O5.**

  **Architecture A — decrypt-and-re-encrypt (bridge is an endpoint).** The bridge holds both crypto sessions. Sender E2E-encrypts to the bridge's Rumor pubkey; bridge decrypts; bridge re-encrypts to the LoRa contact's PKC key. Pure-relay DMs (Rumor → Rumor that happen to traverse a bridge node) stay E2E because the bridge has no key for messages it isn't the addressed recipient of. **Crypto is sound; trust model changes — the bridge operator can read bridged DMs.** Works for any LoRa stack regardless of envelope format. Required UI before ship (HARD blockers, not nice-to-have):
    1. Bridged contacts must render with a distinct "via <bridge>" label everywhere they appear (compose, contact list, thread header). Never as if they were direct Rumor contacts.
    2. The compose screen for a bridged contact must display a persistent banner: "Messages to this contact pass through <bridge owner>'s device and are readable there." No dismiss.
    3. SAS / safety-number verification must include a three-leg fingerprint (you ↔ bridge ↔ them) instead of the two-leg flow used for native Rumor contacts.
    4. Identity rotation: if the bridge node's Rumor key changes, every bridged contact must be re-verified — bridge-key change ≠ peer-key change but UI must treat it as either.

  **Architecture B — envelope passthrough (bridge sees ciphertext only).** Rumor's DM envelope for bridged contacts becomes byte-compatible with the target network's DM envelope. The bridge forwards opaque ciphertext both ways and never holds plaintext. Preserves real E2E across the bridge. Curve alignment exists for **both** bridges — Meshtastic PKI uses X25519 directly; MeshCore uses Ed25519 for identity and ECDH from the X25519-converted key, same as Rumor. The differences are only in the AEAD wrapper, not in the curve:
  - Rumor native: X25519 ECDH + AES-GCM
  - Meshtastic envelope: X25519 ECDH + AES-CCM + Meshtastic framing
  - MeshCore envelope: ECDH (Ed25519→X25519) + AES-128-ECB + HMAC-SHA256 (Encrypt-then-MAC) + MeshCore framing

  Requires per-bridge:
    1. A second DM-encryption code path in `GossipEngine.composeDirect` selected by recipient userId prefix (`meshtastic:*` → Meshtastic envelope; `meshcore:*` → MeshCore envelope; native → existing X25519+AES-GCM).
    2. Bridge advertises discovered remote NodeInfo / contacts as synthetic Rumor contacts whose `publicKey` field holds the remote pubkey (32 bytes for both — Meshtastic X25519 directly, MeshCore Ed25519 which we convert at ECDH time).
    3. Rumor's Ed25519 signature is dropped for bridged DMs — neither target envelope carries an outer signature, only an AEAD/MAC tag. Acceptable because authenticity is enforced by the AEAD/MAC plus the recipient's knowledge of which key sent it.

  Default recommendation: **Architecture B for both bridges.** Worth the per-bridge envelope code to preserve real E2E. Fall back to Architecture A only if a future bridge has no usable ECDH path. Both architectures still require the bridged-contact UI labelling from A1 above — even passthrough has a bridge in the path, and users need to know which contacts are bridged so a compromised bridge can't quietly substitute its own key for a peer's.

  **Framework: pluggable `DmEnvelope` (O5a).** To avoid hardcoding "if recipient starts with meshtastic: do X, if meshcore: do Y" into core, route DM crypto through a registered envelope. Each bridge plugin owns its envelope; core only knows the interface.

  ```kotlin
  // core/plugin/DmEnvelope.kt
  interface DmEnvelope {
      val recipientPrefix: String                  // e.g. "meshtastic:"
      val envelopeId: String                       // stable wire-format id, persisted on the message
      val selfAuthenticating: Boolean              // true → engine skips outer Ed25519 sig
      fun encrypt(recipientUserId: String, recipientPubKey: ByteArray, plaintext: ByteArray): ByteArray
      fun decrypt(senderUserId: String, senderPubKey: ByteArray, ciphertext: ByteArray): ByteArray?
  }

  // additions to PluginContext
  fun registerDmEnvelope(envelope: DmEnvelope)
  fun injectBridgedDm(
      recipientUserId: String,
      senderUserId: String,
      senderPubKey: ByteArray,
      ciphertext: ByteArray,
      envelopeId: String,
  )
  ```

  Compose path: `GossipEngine.composeDirect` consults the registry by recipient prefix; falls back to the existing native envelope when none matches. Inbound path: bridge plugin calls `injectBridgedDm` with an opaque blob plus the envelope id; engine routes to the recipient's inbox without ever holding plaintext; the recipient's UI looks up the envelope by id and decrypts at read time.

  This is a prerequisite for O5 if we want more than two bridges without forking core every time. Even with just Meshtastic + MeshCore today it is the right shape — two hardcoded branches in `composeDirect` would be the same wrong direction as the original Hilt → Koin "just rewire it where you need it" pattern we already abandoned. Implement O5a first, then O5 for each bridge is a small, isolated addition.

  **Security constraints for O5a (non-negotiable — without these, the framework is a regression).**

  1. **Source-gated `selfAuthenticating`.** Honored **only** for `MessageSource.LOCAL_BRIDGE` (messages arriving via `injectBridgedDm`). Over peer transport, every message must pass Ed25519 verification regardless of any envelope-id claim. This is the same rule that gates `BRIDGE_UNSIGNED` today (commit f898874); the lesson generalises directly. A peer transport claiming `selfAuthenticating` is treated as a corrupt frame and dropped.
  2. **Envelope id is derived, not asserted on the wire.** Recipient looks up the envelope by recipient userId prefix locally. An envelope-id field on the message exists only as a sanity assertion at the bridge boundary — never as the decryption selector. Prevents downgrade attacks where an attacker injects a weak envelope id to coerce decryption with attacker-controlled assumptions.
  3. **One envelope per prefix; registry is append-only per plugin lifecycle.** `registerDmEnvelope` throws on prefix collision. Plugin teardown unregisters atomically. Prevents prefix-squatting plugins from intercepting another bridge's DMs.
  4. **Bridged DMs inherit BRIDGED trust → never re-relayed.** Same invariant as broadcast bridge traffic today.
  5. **Replay protection is the envelope's responsibility.** Document this as a requirement. Native Rumor envelope uses dedup + sequence; bridged envelopes inherit the target network's mechanism (Meshtastic packet id + timestamp; MeshCore sender_timestamp tracking). A reviewer adding a new envelope must explicitly state how replay is prevented.
  6. **Trust model unchanged from existing plugins.** A user-enabled plugin already has full access to broadcast plaintext via `observeIncoming`. Adding DM handling raises the leakage surface only for users who enable a malicious bridge plugin — same gate (PluginCatalog toggle) as today. The framework does NOT widen the trust boundary; it just gives well-behaved bridges a way to preserve E2E that they previously didn't have.

  If any of these six constraints can't be honored by a proposed envelope, fall back to Architecture A for that bridge.

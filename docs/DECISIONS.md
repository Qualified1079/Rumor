| O11 | **`[DECISION]`** `no-bridged-rerelay` assertion may need escalation** | The scenario runner's `NoBridgedRerelay` check is a post-hoc snapshot scan: at the end of a scenario it walks every node's stored messages and flags any BRIDGED-trust message whose `recipientId` doesn't match that node's `userId`. It's reliable in the simulator because the in-memory `MessageRepository` never evicts during a run, but it would miss a violation where the engine relayed a bridged message, then the message expired or got evicted before the scan. If the in-sim message store ever starts trimming during a scenario — or if this assertion graduates to production telemetry — escalate to a live observer: add a `relayObserver: ((RumorMessage) -> Unit)?` hook to `GossipEngine.relay()` and have the runner subscribe per-node. The current implementation is in `simulator/.../scenario/ScenarioRunner.kt` and is intentionally marked temporary in the JSON-schema docstring on `Assertion.NoBridgedRerelay`. |
| O12 | **`[DECISION]`** `deterministic-replay` residual non-determinism** | `SimTransport.exchange()` was fixed to (a) drop real `delay()` calls and (b) use the tick's seeded RNG. Two variance sources remain: (1) `GossipEngine` runs internal handlers on `Dispatchers.Default`; coroutine scheduling order isn't guaranteed (affects `totalMessages` by ~3–5%). (2) `ConcurrentHashMap` iteration order in `knownMessages()` determines which RNG values the conditioner consumes; drop counts vary by <0.1% across runs. Both are tolerated by ±5% relative bounds on both messages and drops in the assertion. If variance grows, the escalation path is a single-threaded dispatcher for `SimNode.scope` (to fix handler order) and sorting `knownMessages()` by ID (to fix conditioner RNG consumption order). Neither done preemptively — the 5% band keeps the assertion useful as a regression catch. |
| O63 | **`[DECISION]`** iOS via shared `:core` (KMP), not a separate app** | Separate-but-interoperable iOS app rejected: two protocol implementations diverge silently, and the wire is the only place a bug is unrecoverable post-ship. Four hard constraints, in order — do not skip ahead: (1) **stable storage interfaces in `:core` before any SQLDelight migration** — DAOs (Room today, SQLDelight tomorrow, Core Data never) must not leak past the adapter layer; anything in `:core` importing a DAO type is the bug. (2) **`:core` compiles unchanged on JVM, Android, Linux, iOS** — platform deltas behind `expect/actual`; the simulator already proves the storage interfaces are honest. (3) **plugin loading needs a declarative (non-code) tier before ecosystem ships** — DEX-only is Android-only by construction; iOS forbids runtime code load entirely. (4) **NO Android feature regression when iOS lands** — iOS may ship a strict subset, negotiated *down* per session via HELLO `supportedFeatures`, never enforced upward onto Android; reviewer test for every cross-platform change: "does this still work for two Android peers?" If no, reject. (1)+(2) are worth doing regardless of iOS — they also unlock O54/O106 Linux nodes. **Post-merge note (2026-07-16):** the KMP layout was flattened to plain JVM in the merge; `core/platform/` shims + `ios/` bridge spec kept so this decision stays cheap to execute — re-splitting is per-file mechanical. Full per-constraint elaboration + iOS-specific work list archived in `docs/COMPLETED_GAPS.md` (O63 archive). |
| O82 | **`[DECISION]`** Median-clock consensus — NOT a core protocol feature; each use case has a better local fix | The proposal was Bitcoin-style network-adjusted-time at HELLO so code that wanted "a shared clock" could read a median across recent peers. **Rejected on review:** every use case Rumor actually has is better served by a local primitive that doesn't require trusting peer-reported clocks. (a) Relay-deletion-on-ACK (O40) → TTL since `receivedAtMs`; each relay decides locally. (b) Scheduled messages (O22) → fire on sender's own clock; no consensus needed. (c) Room mod-action ordering (O79) → Lamport / vector clocks (per-node monotonic counters), the textbook distributed-systems answer. (d) Anti-replay windows → `receivedAtMs` delta or sequence-number window. (e) "What time is it?" without internet → real UX question but it's a UI/utility need, not protocol; ships as an opt-in plugin or sister app (think mesh-NTP-replacement), never in `:core`. The median-clock approach is sound math chasing a problem that doesn't exist here — and it adds Sybil surface (peers can lie about their clocks to drag the consensus) for no actual win. Row kept as a decision so the temptation to add it recurs against the recorded rationale.  Feeds the O95 ordering decision as an input. |
| O86 | **`[DECISION]`** Payments are out of Rumor scope — full stop (user ruling 2026-07-19) | A "mesh crypto" / value rail (Cashu, Lightning, generic) is useful when external infrastructure is degraded, but it is NOT Rumor's job and NOT a companion we build — a DEX or external plugin does this. Rationale: money-transmission regulation (FinCEN MSB / EU MiCA) may attach to whatever "facilitates" transactions, so Rumor stays unambiguously a transport carrier; and conflating identity/messaging/payments into one audit surface is a liability. Rumor exposes only identity + signing + opaque-blob DM + cross-network bridging (O84); anything wanting to carry value builds on that externally, never inside Rumor. |
| O104 | **`[DECISION]`** Post-SHTF hardware realism: owner-unlockable Androids + USB-bootable Linux, everything else is firmware-locked** | Modern devices are mostly too locked down at the firmware level to repurpose. The realistic post-SHTF hardware inventory is exactly two classes: **(1) Android phones still in their owner's possession** — the owner can unlock and sideload, which is Rumor's current target; **(2) Linux-capable laptops/desktops** that can boot Rumor from USB or be reflashed from USB via the boot menu. Everything else (iOS, smart TVs, consoles, locked-bootloader routers, most Chromebooks) is effectively e-waste for mesh purposes — do not design for commandeering it. **Implication for O54:** deprioritize exotic embedded ports; the second form factor should be a **bootable live-USB Linux image** running a Rumor node. Key simplification: `:core` is pure JVM and `:simulator` already proves it runs on desktop Linux — a Linux node is *packaging + transport* work (wpa_supplicant P2P / mDNS-LAN per O93 / USB link), **not a rewrite**. The "port the relay subset to non-JVM" idea in O54 remains only for genuinely constrained targets, if any survive this filter. Refines O55's fixed-infrastructure framing: window-mounted phone + USB-booted laptop on UPS are the anchor form factors. **Analysis (agreed 2026-07-15):** (i) the Linux node should skip wpa_supplicant Wi-Fi Direct entirely — **the laptop should *be* the AP**: hostapd + a Rumor node on its own LAN, phones join as ordinary Wi-Fi clients, mDNS/LAN transport (O93) carries frames. Strictly easier than making Linux speak P2P, and it *solves* O93's STA-coexistence problem for every phone nearby — any USB-booted laptop becomes exactly the fixed-infrastructure anchor O55 wants. (ii) x86 laptops are even more viable than framed: Secure Boot without a BIOS password still usually permits USB boot; the truly locked class is mostly ARM tablets/Chromebooks. |
| O125 | **`[DECISION]`** Extract/generalize unique code + knowledge into standalone artifacts — DEFERRED until post-wire-lock / first release** | Recurring idea (raised 2026-07-18): pull Rumor's distinctive pieces out as reusable libraries. **Deferred, not rejected — and mostly the wrong vehicle.** Rationale: (a) the strict `:core`/`:app`/`:simulator` compartmentalization already IS the extraction that pays off — `:core` is pure JVM behind storage/transport interfaces, and `:simulator` is a permanent *second consumer* that keeps those interfaces honest (the API-review benefit of publishing, for free, without publishing). (b) Publishing artifacts adds semver discipline, API-stability promises, separate CI/repos, and — the load-bearing objection — **freezes wire/API surfaces that are deliberately still fluid** (e.g. the HLC counter went Int→Long "for free" this cycle precisely because the wire is fleet-only; a published lib kills that). Pre-first-release, extraction is paying interest on a loan nobody's taken out. **When the time comes, candidates are NOT equal — ranked:** (1) **`HlcClock`** — small, pure, genuinely generic, pin-attack analysis already written (G42); cleanest extraction. (2) **`Rbsr.kt`** — strongest on paper (no good Kotlin negentropy exists publicly) but Rumor's fingerprint deliberately diverges from the reference (additive-over-hashed-items, because ids are sender-chosen — see O-rbsr-v2 tombstone), so it's NOT NIP-77-compatible; honest extraction means "negentropy-hardened-for-adversarial-ids" (niche) or a pluggable formula (reintroduces the forgeability footgun) — record this tension before anyone extracts it enthusiastically. (3) The **Wi-Fi Direct work** (`PersistencePlanner`/`BackboneRealizer`/`GroupCredentials`) — the valuable artifact is a **writeup, not a library**: the hard-won knowledge is field facts (only autonomous `createGroup` honors the channel; deterministic SSID/PSK gets prompt-free joins; per-OEM quirks) that transfer as documentation far better than as a Maven dep whose hardcoded parts every other mesh will differ in. **Never extract** the crypto shims (`HmacSha256`/`HkdfSha256`) — the world has audited versions. Tiny generic pieces (`CompressedPaddedCodec`, `MultiRecipientEnvelope`) — vendor ~200 lines, extraction overhead exceeds the code. **Knowledge-extraction corollary:** `docs/wire-format.md` (already an O75 precondition for byte-compatible reimplementations) + the field-fact writeups are the better generalization vehicle for most of this project's uniqueness; specs over libraries. |

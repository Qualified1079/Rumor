| O21 | **`[TODO/UI]`** TOFU key pinning and identity hardening** | `userId` is already `SHA-256(publicKey).toHex()` (see `CryptoManager.publicKeyToUserId`), so the userId↔pubkey binding is cryptographic — a sender cannot claim a userId that doesn't hash from their pubkey, and every receiver can re-verify cheaply. The real identity-hardening gap is **display**: a 64-char hex userId is unreadable, and human-set display names can be freely spoofed (two people can both call themselves "Bob"). Solution: derive a short visual fingerprint from the pubkey — take 8 bytes of `SHA-256(publicKey)`, map each byte to an emoji from a curated set of ~256 visually distinct emoji (animals, food, objects, vehicles; no faces, no flags, no near-duplicates, all cross-platform-stable). Display as `Bob [🐅🍧🚲🌵]`. Four emoji = 32 bits of fingerprint, collision-resistant up to ~65k contacts, easy to compare at a glance. Full hex userId revealed only on tap-through. **Pin display-name → userId binding** in `ContactRepository`: if a previously-seen userId arrives with a different display name, surface "Bob changed their display name to Robert" rather than silently updating. Pinning is for display-name churn, not key churn (latter is impossible by construction). Display names should be entirely optional — users who want maximum clarity can show only the emoji fingerprint. |

| O69 | **`[TODO/UI]`** In-app full-wipe ("forget me") flow** | App Store 5.1.1 requires in-app account deletion. Rumor has no cloud account, but the UI must still expose a one-tap "delete identity and all data": wipe Ed25519 keypair, Room DB, SharedPreferences, transfer cache, log ring buffer, plugin scoped storage. Double-confirm with a 5-second undo timer before commit. **Required: emit a signed `IDENTITY_RETIRED` broadcast before wiping**, signed by the about-to-be-deleted Ed25519 key, telling contacts the identity is retired. Recipients update the contact record to "retired" status (display greyed, no further messages expected from this userId, blocked from re-establishing under the same userId since the holder is supposedly gone). | O18 | **`[PART]`** Transfer-control UI (pause / resume / cancel)** | `TransferAssembler` and `TransferSender` have no user-facing controls. Once a large transfer starts, the user can't cancel it without quitting the app, and it can't be paused if the device is going to be off-network temporarily. Add: cancel button (drops the transfer record, cancels watchdog, sends a CANCEL message so sender stops resending chunks), pause button (suspends watchdog and incoming chunk processing without dropping state), resume (re-arms watchdog from current state). UI should show current progress (`receivedIndices.size / totalChunks`), throughput, and estimated time remaining. No protocol changes for cancel/pause if implemented as receiver-local; sender-side cancel needs a new `TRANSFER_CANCEL` message type so the sender stops queuing chunks. |
| O14 | **`[TODO/UI]`** Compressed-archive extraction UI** | There is no compression at the protocol layer — `Chunker` splits raw bytes, no zip/gz. So a zip bomb at protocol level is not possible; the hash is of the full uncompressed data and is covered by Ed25519, so a sender cannot lie about it without breaking their signature. The risk is purely application-layer: a sender transmits a 1MB `.zip` payload which the receiver extracts to 100GB. Fix: before any extraction of a compressed MIME type, show a non-dismissable dialog: "Compressed archive received — Download: XMB / Estimated extracted: YGB." Do not extract until user confirms. This is a UI feature, no protocol changes needed. |
| O19 | **`[TODO/UI]`** Pre-send file manipulation (compression, format conversion)** | Pure app/UI layer, no protocol changes. Before chunking and sending a file, offer the user: image compression (JPEG quality slider, downscale to N pixels), video transcode (bitrate selector via Android's `MediaCodec`), archive options for multi-file selections (zip / tar.gz). Show "Original: XMB → Compressed: YMB" before confirm. Reduces bandwidth and storage cost across the mesh significantly for image-heavy traffic, which is the dominant case in practice. Independent of O14 (extraction-warning popup) but shares the same UI surface. |

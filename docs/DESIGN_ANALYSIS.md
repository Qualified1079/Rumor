| O77 | **`[TODO/SIM]`** Massive long-uptime scenario (500 nodes, 24-hour sim, sustained churn) | Today's scenarios cap at 200 nodes / 120 s sim-time and never trigger storage eviction, bloom degradation at scale, or the kind of sustained-churn protocol exercise the O55 prolonged-disconnection design target requires. Predicted-but-unproven behaviors that only a massive sim can surface: (a) `MessageRepository.evictOldest` actually fires — does the eviction policy keep a coherent working set or churn the freshest content out? (b) Static bloom `falsePositiveRate` against per-node sets of 10–60k messages — does over-offering wreck bandwidth? (c) `MAX_MESSAGES=50000` per node × 500 nodes ≈ 12 GB worst-case heap — does the storage policy hold? (d) Multi-hour partition heal with churn during the partition — does anything converge? (e) Hub bottlenecks at scale — at what node count does hub-spoke fairness collapse under queue pressure? **Recommended scenario shape:** 500 nodes, static mesh topology with k=6 connections per node, 24 sim-hours, gentle steady-state traffic (1 broadcast/node/10 sim-min ≈ 72k broadcasts total), rolling partitions every ~30 sim-min healing 10 min later, breadcrumbs ON, RBSR OFF (until per-round cost optimization lands — G17). **Assertions:** heap stays under 4 GB; per-node message count stays under 60k after eviction kicks in; cross-partition delivery ratio stays >0.6; no node crashes; deterministic-replay within ±5% (now that O12 clock injection is in place, this should hold even at scale). **Operational requirements:** wall-clock cap raised from 240 s default to 1800 s (30 min); JVM `-Xmx16g`; bundle file will be ~70 MB (one per scenario); single scenario per run, not bulk sweep. Probably needs a dev box with ≥32 GB RAM to run comfortably. Deferred until a fresh hardware target exists — phone-class dev machines won't fit. When run, hand `result.json` summaries to a reviewer; the full trace is too big to read into chat context. Probably surfaces 1–3 real protocol issues invisible at 200-node scale — that's the point. |

| O101 | **`[TODO/CODE]`** First-class signed sites/pages** | Users already build "site-analogues" on every messenger (Telegram/Signal/Instagram bots) — clunky and inefficient; a real page primitive is better UX. **Bandwidth must be designed up-front** (popular sites get hammered): (1) **easy cloning/mirroring** of a whole site or individual pages; (2) **author-signed site + pages** so a mirror is verifiably the author's work and a cloned-site-with-malware is detectable; (3) **aggressive local rendering** — rendering machinery (HTML scaffolding, styling) lives on the client; the wire carries compact content only; (4) **constrained format** — no heavy JS; even heavy markdown/HTML is non-negligible at Wi-Fi Direct bandwidth; (5) **RSS-like subscription feed shipping diffs**, not whole pages — and diffs are **cached at intermediaries** so a request needn't travel to the source; (6) users can **mirror pages they value** themselves (signed, so provenance survives). **Discovery** ("what sites even exist?"): a **"business card"** — bio + the sites you host or mirror (each with its author signature) — exchanged as an **optional post-HELLO phase, never inside HELLO itself** (the signed handshake transcript must not accrete weight). Stored once; every later session asks only **"what changed?"**. Advertising what you host/mirror is an O58 Tier-3-style visibility disclosure — **opt-in per site**. **Per-site storage scopes:** hosting a wiki must not also expose your furry art page — each site is its own share scope, opt-in per site, while everything on a site remains shareable (extends O23's quota/scope model). O36 compliance: sites are author-signed, subscription/pull content inside the contact-graph model — *not* an open room; discovery is peer-exchanged business cards, never a network-level public directory. Cross-refs: O100 (big site assets ride swarm fetch), O42 (diff sync machinery), O21 (author fingerprint display on business cards). **Analysis (agreed 2026-07-15):** (i) **Format decision — no HTML, no WebView, ever.** An HTML subset inherits sanitization hell and XSS-in-a-WebView as a permanent security tax; a constrained markdown-ish format rendered **natively in Compose** kills the whole vulnerability class, enforces the bandwidth budget structurally, and satisfies no-CDN/offline-first by construction. This is THE scoping decision — make it first, before any other design work on this row. (ii) **Architecture — a site is a broadcast feed with structure.** Signed page-versions are messages; the diff feed *is* gossip; intermediary caching *is* the message store. Modeled that way, this builds almost no new distribution machinery — mostly a native renderer + subscription UI. |

| O103 | **`[TODO/CODE]`** Geotag / collaborative map-annotation plugin** | Post-SHTF, digital maps go stale: OSM files can be *transferred* (O100 helps) but not collaboratively *edited* — "this bridge is out", "this building burned down", "this place is a clinic". Plugin: **signed geo-annotations** gossiped as ordinary signed content; merge is append-only per (author, location), display renders latest-per-author with conflicts visible (no consensus needed — annotations are attributable claims, and O21 fingerprints tell you whose claim you're trusting). User judgment: genuine value-add post-SHTF, hopefully not out of scope — scoped as **plugin-tier** (O15/O24), not core. **Platform note (recorded):** plugins deepen the iOS lock-out — Apple forbids post-install code, so the DEX-loader model is un-portable; iOS was already excluded by the APK/DEX architecture, but if iOS ever matters, high-value plugins like this become candidates for core-app inclusion instead of plugin distribution. **Analysis + user correction (2026-07-15):** (i) **No automated decay/expiry UX — deliberate design decision.** Post-SHTF wall-time is untrustworthy (O95 — the 64-day-slow field phone would have back-dated every annotation), so staleness must never be machine-resolved. Instead **annotations are commentable**: "bridge out" accrues signed replies — "it's back", "no it isn't" — which are themselves annotations anchored to the parent, and *people* decide what to believe. Timestamps are displayed honestly but never drive auto-hide. (ii) Map renderer is a **must-adopt** per the prebuild policy — osmdroid or MapLibre (FOSS, F-Droid-compatible), never hand-rolled. (iii) This is the strongest single post-SHTF demo of why Rumor matters ("the bridge is out" on a map beats any messaging demo) — **core-app candidate**; plugin-tier is the right first home, expect promotion if it proves out. |

| O102 | **`[TODO/SIM]`** Neighbor-aware forward delegation (one copy per airtime)** | If A, B, and C all want content from F, and F's only neighbor is E, naive delivery has E sending the same bytes in triplicate. Since E knows A/B/C are mutual neighbors, it can send **one copy to A with the instruction "forward to B and C"** — the payload occupies one link at a time. **User's own caveat:** pure single-file serialization could *slow* the network; combine with **minimal-overlap coverage sets** (person 1 sends to 2,3,4,5; person 6 sends to 5,6,7) — and note the existing P2 overlap-shaped batching may already approximate this ("maybe this is already the solution?" — test, don't assume). Physical-layer nuance: within one Wi-Fi Direct group all transmissions share the channel anyway (CSMA), so the win inside a group is airtime, and the big win is multi-group/multi-hop. Security: a delegation instruction is a **work order from a peer** — must be budgeted (trafficClass ceilings, relay caps) so a malicious node can't use it for amplification. Deliverable: simulator comparison — naive N-copies vs delegation vs P2-overlap-only vs combined — measuring airtime, latency, per-node bandwidth. Cross-refs: P2, O29 breadcrumbs, O58 Tier 1/2, O43, O100 (a swarm naturally load-spreads; delegation attacks the same waste from the sender side). **Analysis (agreed 2026-07-15) — expected outcome negative:** (i) physics: within one Wi-Fi Direct group all nodes share the channel (CSMA), so E→A + A→B + A→C moves the same bytes over the same air as three unicasts from E, *plus* instruction overhead — delegation only wins across multiple groups/hops with genuinely parallel channels; (ii) gossip already performs implicit delegation self-healingly (E hands to A, A's next exchanges cover B and C, dedup prevents the triplicate) with no trust surface and no silent failure mode — an explicit instruction dies with A's battery and nobody retries, because E believes it's handled. Prediction: the sim shows a marginal win for messages, and O100's swarm covers the bulk-transfer case. Run the cheap sim experiment anyway; if confirmed, close this row as a DECISION so the idea doesn't recur. |
